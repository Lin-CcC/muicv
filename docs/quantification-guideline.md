# 简历语境下「量化」的精确定义

> 本文档为权威源。所有引用「量化」概念的 skill（muicv-core / muicv-generate /
> muicv-critique / muicv-debrief / muicv-audio-review / muicv-interview）应回写
> 一段一致的精炼摘要，并指向本文档；摘要做到"复制粘贴一致"，便于日后 `grep -lr`
> 同步。

## 为什么要专门定义这个词

简历里到处都在讲"量化"，但 LLM 在收集素材或生成文本时容易把**任何带数字的描述**
都当成"量化"来填。**这是个高频 bug**：

> 用户说"这个项目我们覆盖了美/德/日 3 个站点"——LLM 把"3 个站点"当作 Bullet 的
> "量化"成绩列出，实际上这是**范围 / 规模描述**，不是简历招聘官在看的那种"量化"。

招聘官看的"量化"，是用来回答**"这件事你做成了多大效果"**——不是"你做了多少件事"。

## ✅ 是「量化」（outcome metric）

简历语境下"量化"必须是 **Action 之后产出的成果指标**，即"做成了什么、改善了多少、
效率/质量/规模/收益变化"。常见 4 类：

### 产品 / 增长
- 转化率 +12%、点击率 ×2.3、注册→付费转化率从 4% 升到 9%
- LTV 提升 30%、ARPU 从 ¥45 升到 ¥72
- 留存（D7）从 22% → 35%、流失率减半

### 运营 / 业务
- 月销 ¥80k → ¥260k、GMV 同比 +180%
- 客单价 +¥35、退货率从 8% 降到 3%
- 品类排名 #1（在 N 站点中，配合"促成"的事实陈述）
- 复购率 +20pp、库存周转天数 28 → 14

### 工程 / 性能 / 质量
- P75 加载时间 800ms → 320ms、TTFB 降 40%
- 构建 12min → 4min、CI 失败率 -60%
- bug 率从 0.8/千行降到 0.2/千行
- 回滚频率 / MTTR 减半

### 管理 / 组织（注意：人数本身不算，要绑结果）
- 团队从 4 人扩到 12 人 **且** 同期 ARR 从 ¥X 涨到 ¥Y——绑产出才算量化
- 招聘周期从 60 天压到 28 天
- 离职率从 22% 降到 6%

**判别标志**：这个数字本身是"前→后变化"或"达成的结果指标"，去掉它整段叙述就缺了
**结果**这条腿。

## ❌ 不是「量化」（scope / count / duration）

下面这些都是简历里常被误标成"量化"、但其实只是**做事的范围、编制、时长、产出件数**：

- "覆盖 **3 个站点** / **5 个国家** / **7 条产品线**"——是 scope（范围）
- "带 **4 人团队**" / "Manage **6 个 PM**"——是编制
- "做了 **10 期周报**" / "发布了 **30 篇技术博客**"——是产出**件数**而非效果
- "维护 **30+ 个仓库**" / "owned **12 个 microservice**"——是工作面
- "用 **5 种语言** / **8 个框架**"——是技术广度
- "做了 **3 年**" / "在岗 **5 年**"——是时长

这些数字**可以保留**——它们是事实——但不要把它们填进"量化"字段或独立标"量化:"。
**正确做法**：把它们写进 Action 描述里，作为"在多大范围/用什么资源做了什么动作"
的一部分。

> ❌ 反例（bug 形态）：
> ```
> Bullet: 多站点差异化策略 → 德国站 Find My 品类排名第一
> - 核心叙事：针对美/德/日 3 个站点分别制定策略...
> - 量化：覆盖 3 个站点
> ```
>
> ✅ 修正：
> ```
> Bullet: 多站点差异化策略 → 德国站 Find My 品类排名第一
> - 核心叙事：在美/德/日 3 个站点分别制定差异化策略（德国主推 Find My 场景）
> - 量化：德国站 Find My 品类排名 #1（如果素材里还有 GMV/转化变化，一并补上）
> ```
>
> 即"3 个站点"消化在叙事里，不再独立升格为量化；真正的量化是**排名 #1** 这个**结果**。

## 三个判别问题（在你想标"量化"之前问自己）

1. **这个数字描述的是"做了多少"还是"做出了什么效果"？** scope 就退回 Action。
2. **这个数字前后能不能配 ↑ / ↓ / → 的变化？** 不能 → 大概率不是量化。
3. **去掉这个数字，叙述还能完整传达"成果"吗？** 还能 → 数字是装饰、不是 outcome。

## 通用兜底规则

- **素材里只有 scope 类数字，没有 outcome**：保留定性表达（"显著提升"/"明显改善"），
  **不要为了凑量化而编数字**，也**不要把 scope 升格成量化**。
- **素材里有 outcome 数字**：原文保留，不要改大改小、不要凑整。
- **从没出现过的数字**：禁止编造（这是 muicv-generate 的核心约束）。

## 评审统计口径（muicv-critique 用）

"量化比例"分子**只算 outcome 数字**：% / 倍数 / 前后对比 / 收益金额变化 / 排名（绑结果时）。
**显式排除**：单纯计数（站点 / 团队人数 / 产品 / 仓库 / 篇数）、时长（年/月）、
技术广度（用了 N 种语言）。

如果一条 highlight 把 scope 数字独立标成"量化"，给一条 P1 建议：**改回 Action 的一部分，
找出真正的 outcome 写到结果位**。
