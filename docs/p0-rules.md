# P0 规则：muicv 简历严重问题判定与修正

> 本文件是 muicv 体系里"什么算 P0 严重问题"的**唯一权威定义**。
> `muicv-generate` 在写文件前的自检循环、`muicv-critique` 在评审时的 P0 维度，都引用这一份。
> 改判定标准只改这里。

P0 = 影响录用决策的硬伤。出现 P0，HR / hiring manager 第一眼就能扣分，甚至直接判否。

P0 共三类：**空话**、**编造**、**关键词重缺失**。规则尽量机械可判定，避免主观。

---

## 1. 空话（SPEECH）

### 判定

一条 highlight **同时**满足以下两条 → 命中 SPEECH：

1. 以下列空心动词开头：「负责」、「参与」、「协助」、「帮忙」、「支持」、「配合」、「完成」（独立成句时）、"Responsible for"、"Participated in"、"Helped"、"Assisted"
2. 整条里**找不到具体动作**（怎么做的）**或具体结果**（带来了什么）

辅助命中（弱信号，需结合上下文）：

- 纯形容词描述："一个关键的项目"、"重要的系统"、"复杂的业务"——没有任何动词
- 套话短语："认真负责"、"团队合作"、"沟通能力强"、"快速学习"

### 判定示例

| highlight | 命中 | 理由 |
|---|---|---|
| 负责前端开发工作 | ✅ | 空心动词 + 无动作 + 无结果 |
| 参与了 dashboard 项目 | ✅ | 空心动词 + 无具体动作 |
| 主导 dashboard 重构，将首屏 P75 从 800ms 降到 320ms | ❌ | 实动词 + 动作 + 结果 |
| 负责架构选型，在 React / Vue / Svelte 中评估后选用 React，理由是团队栈一致 | ❌ | 虽以"负责"开头但有具体动作和决策依据 |
| 一个关键的内部工具 | ✅ | 纯形容词，无动词 |

### 修正动作

- 回到对应素材文件（`source_files` 里指向的 `experience/*.md` 或 `projects/*.md`），找该 highlight 的原始描述
- 用下列**实动词**替换空心动词：主导、带领、推动、设计、实现、搭建、重构、迁移、落地、拆分、整合、规范化、优化、降低、提升、精简、收敛、提速、评估、选型、引入、制定、复盘
- 补出"做了什么动作"和"带来什么结果"。**素材里有什么写什么**，没有结果就只写动作，不要编造结果
- 如果素材里就只有"负责 X"四个字，没有更多事实可挖 → **保留原状**，但标记为 partial 通过，在交付摘要告诉用户："这条素材本身就是空话，建议补具体动作和结果（用 muicv-core 的 update-experience）"

## 2. 编造（FAB）

### 判定

任何**简历里出现但素材里查无对应**的事实 → 命中 FAB。重点检查：

1. **数字**：% / 倍数 / 前后对比 / 金额 / 排名 / 用户数 / QPS 等。逐个数字回到 `source_files` 里 grep，找不到精确对应或合理推导链 → 命中
2. **能力声称**："精通 X"、"5 年 Y 经验"、"主导过 Z 项目"——X / Y / Z 在素材里没出现 → 命中
3. **职责放大**：素材写"参与"，简历写"主导"；素材写"协助调试"，简历写"独立解决"——命中（包括把 scope 数字升格成 outcome 量化也算编造，例："覆盖 3 个站点"被写成"使 3 个站点性能提升"）
4. **技术栈虚标**：JD 要求的技术栈，素材没用过，简历里硬塞一条"了解 X" / "熟悉 X" → 命中

### 判定示例

| 素材 | 简历 | 命中 | 理由 |
|---|---|---|---|
| "重构后页面更快了" | "重构后首屏速度提升 30%" | ✅ | 30% 凭空 |
| "优化 3 个站点的构建流程" | "覆盖 3 个站点的优化使构建提速" | ❌ | 3 是 scope，不是 outcome；表达没有把它升格成"提速 3 倍" |
| "优化 3 个站点的构建流程" | "构建提速 3 倍" | ✅ | 把 scope 数字 3 升格成 outcome 倍数 |
| "参与 dashboard 项目，写了几个页面" | "主导 dashboard 项目" | ✅ | 职责放大 |
| profile 没提 Golang | "了解 Golang" | ✅ | 技术栈虚标 |
| "跟后端联调 GraphQL 接口" | "熟悉 GraphQL" | ❌ | 用过即可写"使用 GraphQL"，但"熟悉"是程度词，建议改成"在 X 项目中使用 GraphQL 完成接口联调"更稳妥 |

### 修正动作

- **数字编造**：删除该数字，回退到定性描述（"显著降低"、"明显提升"）。**不要**换一个看起来合理的数字
- **能力声称编造**：删除整条 highlight 或改写为素材里实际的事实（"用过"而非"精通"）
- **职责放大**：动词降级到素材里实际的级别（素材"参与"→ 简历也用"参与"或更具体的"实现 X 模块"）
- **技术栈虚标**：直接删除该条；不要"考虑"、"了解过"地兜底

修正后**重新跑一次 FAB 检测**确认这一条已干净。

## 3. 关键词重缺失（KW-MISS）

### 判定

仅在有 target 时检查。

1. 从 `targets/<slug>.md` 的 JD 正文抽**核心关键词**：
   - 标题里出现的（例："Senior Golang Engineer"→ Golang）
   - 要求列表（"Requirements" / "必备技能"）前 3 项里出现的
   - 正文里出现 ≥ 2 次的技术名 / 职责动词
2. 对照简历正文（不含 frontmatter），grep 每个核心关键词
3. 一处都没有 → 命中 KW-MISS

允许同义命中：JD 写 "Golang"，简历写 "Go" 算覆盖；JD 写 "性能优化"，简历写 "首屏提速"算覆盖。

### 判定示例

| JD 核心关键词 | 简历正文 | 命中 |
|---|---|---|
| Golang | 全文未出现 Go / Golang | ✅ |
| Golang | "用 Go 实现 RPC 服务" | ❌ |
| 性能优化 | "首屏 P75 800ms → 320ms" | ❌ |
| Kubernetes | 全文未出现 K8s / Kubernetes | ✅ |

### 修正动作

- **先回素材库再找一次**：可能在 generate 的相关度排序时被裁掉了。命令：grep `<关键词>` 在 `experience/`、`projects/`、`skills.md` 里
- 找到 → 把对应 highlight 加回简历（即便相关度评分低，关键词覆盖优先）
- 真的找不到 → **不要在简历里编造**。改为：
  1. 在 frontmatter 加 `missing_keywords: [<关键词1>, <关键词2>]`
  2. 在交付摘要里诚实告诉用户："JD 要求 X，素材里没有相关经历，未在简历中虚构。建议用 muicv-core 补充相关素材或评估这个岗位是否匹配"

## 总结：generate 自检循环用法

```
对生成的简历草稿，逐条 highlight 跑：
  1. SPEECH 检测（空心动词 + 无动作/结果）
  2. FAB 检测（数字/能力/职责跟素材比对）
  3. KW-MISS 检测（仅有 target 时，JD 核心关键词覆盖）

命中 → 按本文件"修正动作"小节执行
最多重试 1 轮
未通过 → frontmatter 标记 self_check: partial + 摘要诚实说明
```

## critique 评审用法

P0 维度直接套用本文件的判定规则。critique 报告里 P0 段落的"现状 / 建议"两栏，分别对应本文件的"判定示例"和"修正动作"。
